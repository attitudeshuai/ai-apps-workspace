# 分析技能

## 功能概述

这个技能用于分析 session-4 项目的模型执行结果是否完成了 prompt 要求。

它负责：收集上下文、调用 `implementation-reviewer` agent 执行分析、将结果写入评价文件。

它不负责：继续实现功能、修复代码、帮模型补完未完成部分。

## 两种分析模式

根据 `<类型>` 自动选择模式：

- **源码模式**（默认）：适用于 `0-1代码生成`、`Bug修复`、`Feature迭代`、`代码重构`、`工程化` 等会真实产出代码改动的类型。
  - 只读取提示词文件中的「用户第 N 次提示词」和「模型第 N 次回答 trae session id」
  - **不读取「模型第 N 次回答内容」**，用户也不会提供
  - 直接对比子仓实际源码与提示词，判断改动是否吻合需求、实现是否合理、有没有更好的做法
- **会话模式**：适用于 `代码理解`、`代码测试`。
  - 源码可能没有改动，需要从「模型第 N 次回答内容」中提取分析/测试结论
  - 用户会继续在对话上下文中提供「模型第 N 次回答内容」（附件或粘贴）
  - 沿用旧的判定流程

## 执行前提

执行前必须先向用户确认以下信息：

1. **项目名**，例如：`label-01968`
2. **提示词编号**，即文件名中的 `<类型>-<index>` 部分，例如：`bug修复-01`、`0-1代码生成-04`
3. **第几次对话**，例如：第 1 次、第 2 次、第 3 次

没有项目名或提示词编号时，不要自行猜测，不要直接开始分析。

如果用户只给了项目名，先列出 `02.work session/session-4/ai-model-result/<项目名>-plus/` 下所有类型目录及文件，供用户选择。

如果用户给了项目名和编号，先读取对应提示词文件，列出其中存在内容的对话轮次（「用户第X次提示词」非空的），再询问用户要分析哪一次。

## 输入来源

拿到项目名、提示词编号和对话轮次后：

**源码模式**只读两类来源：

1. `02.work session/session-4/ai-model-result/<项目名>-plus/<项目名>-<类型>/A-<项目名>-<类型>-<index>.md` —— 只取「用户第 N 次提示词」原文 +「模型第 N 次回答 trae session id」。**不读「模型第 N 次回答内容」**。
2. `02.work session/session-4/gitlab source/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>/` —— 读取该子仓实际代码状态，作为分析的核心依据。

**会话模式**读三类来源：

1. 同上，但额外读取「模型第 N 次回答内容」原文。
2. 子仓代码（可能没有改动）。
3. 用户在对话中追加提供的会话补充内容（如有）。

**文件与仓库必须严格对应**：提示词文件和代码子仓的 `<项目名>`、`<类型>`、`<index>` 三段必须完全一致。读取代码前，先把要读取的子仓路径完整写出来，逐段比对确认三段全部匹配后再继续。

## 执行流程

1. 先向用户询问项目名。
2. 列出 `02.work session/session-4/ai-model-result/<项目名>-plus/` 下所有提示词文件，供用户选择提示词编号（类型 + index）。
3. **必须主动调用工具读取**对应提示词文件 `A-<项目名>-<类型>-<index>.md` 的最新内容，不得依赖对话上下文中已存在的附件或缓存内容——文件可能在附件展示后被用户更新。
4. **提示词文件可能非常长（超过 1000 行），不能从固定行号偏移读取来判断轮次是否有内容。** 必须先用 grep_search 搜索 `用户第.*次提示词` 获取所有轮次标记的行号，再根据行号范围判断每个轮次是否有实际内容（即标记行的下一行到下一个轮次标记之间有非空文本）。从读取结果中列出存在内容的对话轮次，向用户确认要分析第几次。
   **重要：提示词内容可能在标记行的下一行开始，而不是写在同一行**，判断轮次是否有内容时必须检查标记行**之后**直到下一个轮次标记之前的内容，不能只看标记行本身是否为空。
5. 确定轮次后，用 grep_search 定位「用户第 N 次提示词」所在行号，提取 prompt 原文，以及对应的「模型第 N 次回答 trae session id」。
6. 根据 `<类型>` 选择模式：
   - 类型为 `代码理解` 或 `代码测试` → **会话模式**：再读取该轮次的「模型第 N 次回答内容」全文。
   - 其他类型 → **源码模式**：跳过「模型第 N 次回答内容」，直接进入第 7 步。不要主动询问用户要不要提供会话内容。
7. 读取 `02.work session/session-4/gitlab source/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>/` 下的源码文件。源码模式必须遍历主代码目录把核心文件全部读完；会话模式按需读取。
8. **将以下内容整体交给 `implementation-reviewer` agent 执行分析**：
   - 这是第几次对话（用于 agent 选择挑剔模式还是实事求是模式）
   - 当前模式（源码模式 / 会话模式）
   - 「用户第 N 次提示词」原文
   - 子仓实际代码文件内容（源码模式必备；会话模式按需）
   - 「模型第 N 次回答内容」原文（仅会话模式）
9. 以 agent 的分析结论为准，填写输出格式的各字段，写入评价结果文件。

## 文件修改字段判定

- **源码模式**：由 agent 根据源码实际状态推断核心改动文件（新增类、明显新增方法、UI 入口等），写出涉及的核心文件名。0 个 → "无代码修改"，1 个 → "单文件修改"，≥2 个 → "多文件修改"。
- **会话模式**：扫描「模型第 N 次回答内容」中文件编辑操作（如 `replace_string_in_file`、`edit_file_search_replace`）出现的文件路径去重统计；若模型在思考过程中读取了多个文件，倾向判定为"多文件修改"。

## 输出文件

评价结果固定追加写入与提示词文件同目录的评价结果文件：

- `02.work session/session-4/ai-model-result/<项目名>-plus/<项目名>-<类型>/A-<项目名>-<类型>-<index>-评价结果.md`

**每次分析结果追加到文件末尾，不覆盖已有内容。** 文件不存在时新建。

## 输出格式

每次追加一个完整评价块，**第一行必须是标志行**，标志从评价结果文件名中提取：取 `label-XXXX` 中的数字（去掉前导零）和 `<index>`，格式为 `A-<数字>-<index>`。例如文件名 `A-label-01035-代码测试-13-评价结果.md`，标志为 `A-1035-13`。

```text

# <项目名>-<类型>-<index> 第 N 次对话评价结果

### 标志：
A-1035-13

## Session ID
<从提示词文件中读取"模型第 N 次回答 trae session id"的完整内容>

## 提示词
<从提示词文件中读取"用户第 N 次提示词"的完整内容>

## 文件修改
单文件修改 / 多文件修改 / 无代码修改
修改文件：<文件名1>、<文件名2>（无修改时省略此行）

## 任务完成度
已完成 / 未完成

## 未完成说明
<仅当任务完成度为未完成时填写，40-60字，只写 prompt 要求但没做到的内容>

## 是否满意
满意 / 不满意

## 不满意的点
<仅当是否满意为不满意时填写，40-60 字，只说具体问题，如果有多个问题，选一个最核心的一个问题>

## 过程满意度
<80-100 字，连贯叙述，不用条目标签，语气直接。源码模式聚焦"改动是否合理、是否有更好的方法"；会话模式聚焦执行过程的合理性>

## 下一轮推荐提示词
<仅当是否满意为不满意时填写，40-60 字，聚焦最核心的一个问题，不要直接指出问题所在，用模糊、结果导向的方式描述，让模型自己发现并修复问题>
```

字段规则：
- 任务完成度为"已完成"时，省略"未完成说明"整段
- 是否满意为"满意"时，省略"不满意的点"和"下一轮推荐提示词"整段
- 是否满意为"不满意"时，必须有"不满意的点"和"下一轮推荐提示词"

## 注意事项

1. **每次分析前必须主动读取文件**，不得依赖对话上下文中的附件缓存或之前读取的内容。
2. **分析上下文来源**：
   - 源码模式：提示词文件（仅 prompt + session id） + 子仓源码
   - 会话模式：提示词文件（含模型回答内容） + 子仓源码（如有改动）
3. **源码模式不要去读「模型第 N 次回答内容」**，也不要要求用户提供，专注源码与提示词的吻合度判断、改动合理性以及更好的实现方式。
4. 以 `implementation-reviewer` agent 的分析结论为准，不要在 agent 之外自行重新判断满意度。
5. 语言自然直接，像工程师在说话，不用"值得注意的是""综上所述""总体来看"这类套话。
6. **整体不满意的字数在 40-60 之间**，字数限制很重要
